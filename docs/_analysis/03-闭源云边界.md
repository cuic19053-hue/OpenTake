# Palmier Pro 后端逆向分析报告：认证、生成式 AI 与 OpenTake 自建后端建议

> 结论先行：**Palmier Pro 的客户端是一个"瘦客户端"。所有生成式 AI 能力都通过闭源的 Convex 部署 + Clerk 认证 + Stripe 计费完成；本仓库不含任何服务端实现。** 客户端从不直接调用 fal.ai/Replicate 等模型厂商，连模型目录、价格、参数 schema 都是后端运行时下发的。OpenTake 必须从零自建这整套生成后端。

## 1. 认证流程：Clerk + Convex

### 1.1 配置注入(无硬编码密钥)

`Account/BackendConfig.swift` 从 `Info.plist` 读取三项配置(运行时由打包流程注入,源码与仓库内 `Info.plist` 都**没有**真实值):

| Info.plist key | 用途 |
|---|---|
| `PalmierClerkPublishableKey` | Clerk 可发布密钥 |
| `PalmierConvexDeploymentURL` | Convex WebSocket 部署地址(响应式 RPC) |
| `PalmierConvexHttpURL` | Convex HTTP Action 地址(用于 Agent SSE 流) |

仓库内 `Info.plist`(`Sources/PalmierPro/Resources/Info.plist`)只声明了 URL scheme `palmier://`(OAuth 回调用)和 Sparkle 自动更新源,**不含**任何后端 URL 或密钥。这是正确的做法——OpenTake 应照搬。

### 1.2 登录链路(`Account/AccountService.swift`)

```
AccountService.configure()
  → Clerk.configure(publishableKey, redirectUrl: "palmier://callback", scheme: "palmier")
  → convex = ConvexClientWithAuth(deploymentUrl, authProvider: ClerkConvexAuthProvider())
```

- 登录方式:**仅 Google OAuth**。`signInWithGoogle()` 调用 `Clerk.shared.auth.signInWithOAuth(provider: .google)`,走 ASWebAuthenticationSession,回调到 `palmier://callback`。
- 关键粘合点是 `ClerkConvexAuthProvider`(来自 SPM 包 `clerk-convex-swift`):它把 Clerk 的 session JWT 自动喂给 Convex 客户端。`AccountService` 监听 `convex.authState.values`(`.loading/.authenticated/.unauthenticated`),状态变为 `authenticated` 时执行 `provisionAndSubscribe()`。
- 用户落库:`convex.mutation("users:upsertFromAuth", {email, name, image})`,带 3 次重试。
- 之后建立两个**响应式订阅**(WebSocket 推送,非轮询):`account:get`(用户/套餐/积分)和 `billing:listPlans`。

### 1.3 Token 如何获取与使用

两种消费 token 的方式,这是理解整个架构的关键:

1. **Convex RPC(主路径)**:开发者**看不到** token 的手动处理。`ConvexClientWithAuth` + `ClerkConvexAuthProvider` 在底层把 JWT 注入每个 mutation/action/subscription。所有生成调用都走这条路。

2. **裸 HTTP(仅 Agent 文本流)**:`Agent/Clients/PalmierClient.swift` 是唯一手动取 token 的地方:
   ```swift
   let session = await Clerk.shared.session         // 必须 .active
   let jwt = try await session.getToken()           // 取 Clerk JWT
   request.setValue("Bearer \(jwt)", forHTTPHeaderField: "Authorization")
   // POST {convexHttpURL}/v1/agent/stream  (SSE)
   ```
   错误码约定:`401/unauthenticated`、`402/insufficient_credits`。

**对 OpenTake 的启示**:认证可用任意 OIDC 提供方(自建或第三方)。核心契约只有一条——客户端拿到一个 Bearer JWT,后端用它鉴权 + 关联用量/扣费。Convex+Clerk 只是上游的实现选择,不是必需。

## 2. 生成式 AI 调用:全部走 Convex 函数,而非客户端直连厂商

### 2.1 调用编排(`Generation/GenerationService.swift` + `GenerationBackend.swift`)

完整的生成生命周期是一个**"上传引用 → 提交任务 → 订阅状态 → 下载结果"**的异步流水线,全部经 Convex:

```
1. 上传引用素材 (uploadReference, 三步式):
   convex.mutation("uploads:generateUploadTicket")     → 取得 Convex 存储上传 URL
   POST 字节到该 URL (URLSession.upload)                → 返回 storageId
   convex.action("uploads:commitUpload", {storageId})  → 返回托管后的公开 URL

2. 提交任务 (submit):
   convex.mutation("generations:submit", {model, params, projectId})
                                                        → 返回 {jobId}

3. 订阅结果 (响应式,WebSocket 推送):
   convex.subscribe("generations:byId", {id: jobId})
                                                        → 推送 {status, resultUrls, costCredits, errorMessage}

4. 下载产物:
   URLSession.download(resultUrls[i])                  → 落到本地工程的 media 目录
```

状态机:`queued → running → succeeded/failed`(`BackendGenerationStatus`)。成功后从 `resultUrls` 下载,失败回传 `errorMessage`。

**关键事实:客户端从不接触 fal.ai/Replicate。** 全仓库硬编码的出站 URL 只有这些(已穷举):

| URL | 用途 | 性质 |
|---|---|---|
| `https://api.anthropic.com/v1/messages` | Agent BYOK 直连 Anthropic | 仅文本 LLM |
| `https://huggingface.co/palmier-io/siglip2-base-coreml/resolve/main` | 下载**端侧**语义搜索模型(SigLIP2 CoreML) | 本地能力,非生成 |
| `https://console.anthropic.com/settings/keys` | 设置页帮助链接 | 静态链接 |
| `https://palmier.io/docs`、`developer.apple.com/...` | 文档链接 | 静态链接 |
| `http://127.0.0.1:19789/mcp` | 本地 MCP 桥 | 本机 |

**没有任何 fal/replicate/openai/elevenlabs 等厂商域名。** 模型厂商 API 密钥与调用完全封装在闭源 Convex 后端里。这意味着上游的**后端 API 形态对客户端是不透明的**——客户端只知道一个抽象的 `model id` 字符串和一组 `params`,不知道这背后是 fal 还是 Replicate,也不知道厂商的真实请求结构。

### 2.2 能看到多少请求/响应结构?(逆向收益)

虽然厂商 API 不可见,但**客户端→Convex 的契约被完整暴露**了,这对复刻极有价值。

**请求侧 —— `generations:submit` 的 `params`** 是一个带 `kind` 判别字段的联合类型(`GenerationBackend.BackendGenerationParams`),四种形态的字段已从 `Encodable` 实现中精确还原:

```jsonc
// kind="image" (ImageGenerationParams)
{ kind, prompt, aspectRatio, resolution?, quality?, imageURLs[], numImages }

// kind="video" (VideoGenerationParams)
{ kind, prompt, duration, aspectRatio, resolution?,
  sourceVideoURL?, startFrameURL?, endFrameURL?,
  referenceImageURLs[], referenceVideoURLs[], referenceAudioURLs[], generateAudio }

// kind="audio" (AudioGenerationParams) —— 同时覆盖 TTS / 音乐 / 音效
{ kind, prompt, voice?, lyrics?, styleInstructions?, instrumental, durationSeconds?, videoURL? }

// kind="upscale" (UpscaleGenerationParams)
{ kind, sourceURL, durationSeconds }
```

所有引用素材(首帧/尾帧/参考图/参考视频/源视频)都先经 `uploads:commitUpload` 变成 URL 再传——**后端只吃 URL,不吃字节**(字节直传 Convex 存储)。

**响应侧 —— `generations:byId` 推送的 `BackendGenerationJob`**:
```jsonc
{ _id, status, resultUrls[]?, errorMessage?, costCredits?, completedAt? }
```

**模型目录 —— `models:list`** 是信息量最大的一处泄露(`Generation/Catalog/ModelCatalog.swift` 的 `CatalogEntry`)。后端**运行时下发**每个模型的能力与定价,客户端纯被动渲染:
```jsonc
{
  id, kind, displayName,
  allowedEndpoints[],          // ← 后端内部端点名,泄露了后端按"端点"组织的形态
  responseShape,               // video|images|audio|upscaledImage
  uiCapabilities,              // 各类型的能力矩阵(见下)
  creditsPerSecond{}, creditsPerImage{}, audioPricing, audioDiscountRate{},
  creditsPerSecondUpscale, qualities[]
}
```
能力矩阵示例(`VideoCaps`):`durations[]、resolutions[]、aspectRatios[]、supportsFirstFrame/LastFrame、maxReferenceImages/Videos/Audios、maxTotalReferences、maxCombinedVideoRefSeconds、framesAndReferencesExclusive、requiresSourceVideo…`。`AudioCaps` 里有 `category(tts/music/sfx)、voices[]、supportsLyrics/Instrumental、minPromptLength` 等。

> 这套 catalog schema 是 OpenTake 设计自己 `models:list` 等价物的现成蓝图——能力声明与定价由后端下发、客户端动态适配,是个值得照搬的好设计。

### 2.3 计费

定价计算在客户端(`Generation/Catalog/CostEstimator.swift`,用 catalog 下发的 `creditsPerSecond/creditsPerImage/audioPricing`)只做**预估展示**;真实扣费由后端在 job 完成时算出 `costCredits` 回传。积分模型:`monthlyBudgetCredits + purchasedCredits − spentCreditsThisPeriod`。计费走 Stripe(`billing:createCheckoutSession` / `createTopOffCheckoutSession` / `createPortalSession`,客户端只打开 `checkout.stripe.com`/`billing.stripe.com` 白名单 URL)。

## 3. 哪些能力依赖闭源云、本仓库完全没有服务端实现

我已确认:**仓库内唯一的"server"目录是 `mcpb/server`**,而它只是一个 11 行的 stdio→HTTP 桥(`npx mcp-remote http://127.0.0.1:19789/mcp`),把 Claude Desktop 接到本地运行的 app。**没有任何 `.ts` Convex 函数文件,没有 backend/functions 目录。** 整个服务端是闭源托管的。

### OpenTake 必须自建的部分(按"客户端调用的后端函数"清点)

| 闭源后端函数 | 能力 | OpenTake 是否必建 |
|---|---|---|
| `generations:submit` / `generations:byId` | **图像/视频/音频/放大生成的提交与状态机** | **必建(核心)** |
| `uploads:generateUploadTicket` / `commitUpload` | 引用素材上传 → 公开 URL | **必建**(或换成 S3/R2 预签名) |
| `models:list` | 模型能力 + 定价目录下发 | **必建** |
| `POST /v1/agent/stream` (Convex HTTP Action) | **AI Agent 的 Anthropic 代理 + 计费**(SSE) | 必建(BYOK 可绕过,见下) |
| `account:get` / `users:upsertFromAuth` | 账户/积分状态 | 必建(若要积分制) |
| `billing:*`(Stripe) | 订阅、充值、客户门户 | 可选(自托管可改 BYOK,免计费) |
| `feedback:send` | 反馈+截图上传 | 可选 |

### 本地/端侧已有、**无需**自建的能力(复刻时直接对应到 Rust+FFmpeg)

- **转录**:`Transcription/`(`SFSpeech`/系统语音,端侧)——与云无关。
- **语义搜索**:`Search/` 下载 `SigLIP2` CoreML 模型到本地跑——端侧。OpenTake 在 Rust 侧可用 ONNX Runtime/candle 复刻。
- **时间线渲染/导出/裁剪/压缩**:`TimelineRenderer`、`VideoCompressor`、`VideoTrimExtractor`(AVFoundation)——OpenTake 用 FFmpeg 替代。
- **MCP server**:app 内置(`Agent/MCP/MCPHTTPServer.swift`,本机 19789 端口),生成类工具仍转调 `GenerationBackend`→云。

> 一句话:**编辑逻辑/媒体管线是本地的(可忠实复刻),生成式 AI 是云的(必须自建)。**

## 4. OpenTake 自建生成后端建议形态

### 4.1 架构:一个开源、可自托管的轻量"生成代理"

照搬上游的抽象边界(这是它最干净的设计),但把闭源 Convex 换成一个无状态 HTTP 服务 + 轻量任务队列。推荐 **Rust(axum)** 单二进制,与 OpenTake 的 Rust core 同栈,易于自托管:

```
OpenTake 客户端 (Tauri/Rust core)
        │  Bearer JWT (OIDC) 或 BYOK header
        ▼
┌─────────────────────────────────────────────┐
│  opentake-gen-proxy  (开源, 可自托管)         │
│  ─────────────────────────────────────────── │
│  GET  /v1/models                 模型目录下发  │
│  POST /v1/uploads/sign           预签名上传URL │
│  POST /v1/generations            提交任务      │
│  GET  /v1/generations/:id        轮询状态      │
│  GET  /v1/generations/:id/stream SSE 推送(可选)│
│  POST /v1/agent/stream           LLM 代理(SSE) │
│  ─────────────────────────────────────────── │
│  Provider Adapters:  fal.ai / Replicate /     │
│                      OpenAI / ElevenLabs / …  │
│  Storage:  S3 / Cloudflare R2 (预签名)         │
│  Queue:    内存 + SQLite/Postgres 持久化       │
└─────────────────────────────────────────────┘
```

**关键设计原则(沿用上游)**:
1. **`model id` 是不透明字符串**;`/v1/models` 在运行时下发能力矩阵与定价,客户端动态适配——新增模型无需改客户端。
2. **后端只吃 URL**:素材先经预签名上传到对象存储,生成请求只传 URL。便于直接转发给 fal/Replicate(它们也吃 URL)。
3. **统一 job 抽象**:`{queued→running→succeeded/failed}` + `resultUrls`,屏蔽各厂商的异步差异(fal queue、Replicate prediction、同步 HTTP 各不相同)。
4. **provider 适配器**:每个厂商一个 adapter,把统一 `params` 翻译成厂商请求,把厂商回调归一化成 `BackendGenerationJob`。

### 4.2 Provider 适配示例(fal.ai)

```
POST /v1/generations {model:"fal:flux-pro", params:{kind:"image", prompt, imageURLs,...}}
  → adapter 查 model 路由 → POST https://queue.fal.run/fal-ai/flux-pro {prompt, image_url,...}
  → 拿 fal request_id,落库为 jobId
GET /v1/generations/:id
  → adapter GET fal status → 归一化 → {status, resultUrls:[fal output url]}
```
Replicate 同理(`predictions` API + webhook/poll)。TTS 走 ElevenLabs/OpenAI。

### 4.3 Rust core 客户端接口草案

与上游 `GenerationBackend` 一一对应,放在 OpenTake 的 Rust core(供前端经 Tauri command 调用):

```rust
// crates/opentake-gen/src/lib.rs
#[derive(Clone)]
pub struct GenClient {
    base_url: Url,
    auth: AuthMode,           // JWT 或 BYOK
    http: reqwest::Client,
}

pub enum AuthMode {
    Bearer { token_provider: Arc<dyn TokenProvider> }, // 自建后端 + 积分制
    Byok   { providers: HashMap<String, ProviderKey> },// 用户自带各厂商 key
}

#[serde(tag = "kind", rename_all = "lowercase")]
pub enum GenerationParams {        // ← 直接复刻上游联合类型
    Image  { prompt: String, aspect_ratio: String, resolution: Option<String>,
             quality: Option<String>, image_urls: Vec<String>, num_images: u8 },
    Video  { prompt: String, duration: u32, aspect_ratio: String, resolution: Option<String>,
             source_video_url: Option<String>, start_frame_url: Option<String>,
             end_frame_url: Option<String>, reference_image_urls: Vec<String>,
             reference_video_urls: Vec<String>, reference_audio_urls: Vec<String>,
             generate_audio: bool },
    Audio  { prompt: String, voice: Option<String>, lyrics: Option<String>,
             style_instructions: Option<String>, instrumental: bool,
             duration_seconds: Option<u32>, video_url: Option<String> },
    Upscale{ source_url: String, duration_seconds: u32 },
}

#[derive(Deserialize)]
pub struct GenerationJob {         // ← 复刻 BackendGenerationJob
    pub id: String,
    pub status: JobStatus,         // Queued|Running|Succeeded|Failed
    pub result_urls: Option<Vec<String>>,
    pub error_message: Option<String>,
    pub cost_credits: Option<i64>,
}

impl GenClient {
    pub async fn list_models(&self) -> Result<Vec<CatalogEntry>>;
    pub async fn sign_upload(&self, content_type: &str) -> Result<UploadTicket>;
    pub async fn upload_reference(&self, path: &Path, content_type: &str) -> Result<String>; // 三步式
    pub async fn submit(&self, model: &str, params: GenerationParams,
                        project_id: Option<&str>) -> Result<String>;          // → jobId
    pub fn watch(&self, job_id: &str) -> impl Stream<Item = Result<GenerationJob>>; // SSE 或 poll
    pub async fn agent_stream(&self, req: AgentRequest)
        -> impl Stream<Item = Result<AgentEvent>>;                            // → /v1/agent/stream
}
```

### 4.4 BYOK 方案(自托管的核心卖点)

上游的 BYOK **只覆盖 Agent 文本**(`AnthropicClient` 直连 `api.anthropic.com`,key 存 macOS Keychain;`AgentService.selectClient()`:有 key 走直连,无 key 走 `PalmierClient` 代理 + 扣积分)。OpenTake 应**把 BYOK 扩展到媒体生成**:

- **双模运行**:
  - **托管模式** = 走自建后端,后端持厂商 key + 积分计费(适合发行版/团队)。
  - **BYOK 模式** = Rust core 用用户自带的 fal/Replicate/OpenAI key,**可完全本地直连厂商,无需任何 OpenTake 后端**(适合个人自托管,零运营成本)。
- **密钥存储**:复用上游模式——macOS Keychain / Windows Credential Manager / Linux Secret Service(Rust 用 `keyring` crate);**绝不**写入工程文件或明文配置。`#[cfg(debug)]` 下允许从环境变量读(上游 `AnthropicKeychain` 就是这么做的)。
- **统一 catalog**:即使 BYOK 直连,也保留 `list_models()` 抽象——可内置一份静态 JSON catalog(不需要后端),让 UI 仍由数据驱动、动态适配各模型能力。
- **provider 中立**:adapter 层让"换 fal 为 Replicate"只是改 model id 前缀,客户端零改动。
