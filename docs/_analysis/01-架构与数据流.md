# PalmierPro 架构分析 → OpenTake (Rust core + React + Tauri) 重建指南

> 证据基准：通读 `Sources/PalmierPro` 下 App / Models / Project / Editor / Timeline / Preview / Export / Agent 共 200+ Swift 文件。所有结论标注了源文件路径与关键行为。核心判断：**PalmierPro 是一个"单一可观测状态容器(EditorViewModel) + 帧本位不可变数据模型(Timeline) + AVFoundation 投影层"的架构。这套结构与 Rust core + React 极其契合——领域模型与编辑算法几乎是纯函数式的，可 1:1 移植到 Rust;AVFoundation 投影层是唯一需要换成 FFmpeg 的部分。**

## 2. 核心领域模型(可直接复刻的数据结构)

### 2.1 基础公理:一切以"帧(Int)"为单位
源文件 `Models/Timeline.swift`。**没有任何浮点秒作为存储单位**——帧是整数,fps 是工程级常量,秒只在 import/与源媒体交互时换算。这是整个编辑逻辑可被精确复刻的关键。

### 2.2 三级层次:Timeline → Track → Clip

全部是 `Codable, Sendable, Equatable` 的**值类型(struct)**——这正是 Rust 移植的理想形态。

```rust
// Timeline (Models/Timeline.swift:9-23)
struct Timeline {
    fps: i32,                 // 默认 30
    width: i32, height: i32,  // 画布尺寸,默认 1920x1080
    settings_configured: bool,
    tracks: Vec<Track>,
}
// total_frames = tracks.iter().map(|t| t.end_frame()).max().unwrap_or(0)

// Track (Models/Timeline.swift:25-59)
struct Track {
    id: String,               // UUID
    r#type: ClipType,         // video | audio (轨道级)
    muted: bool,
    hidden: bool,
    sync_locked: bool,        // 默认 true → ripple 时跟随移动
    clips: Vec<Clip>,
    // display_height 不序列化(UI-only)
}

// Clip (Models/Timeline.swift:75-237) —— 系统的核心
struct Clip {
    id: String,
    media_ref: String,        // 指向 MediaAsset.id(非路径!)
    media_type: ClipType,     // 此 clip 在本轨道的角色(video/audio/image/text/lottie)
    source_clip_type: ClipType, // 原始媒体类型(用于 UI 配色)
    start_frame: i32,         // 时间线上的起点
    duration_frames: i32,     // 时间线上的占用长度
    trim_start_frame: i32,    // 源媒体入点偏移(源帧)
    trim_end_frame: i32,      // 源媒体出点偏移(源帧)
    speed: f64,               // 1.0 = 正常
    volume: f64,
    fade_in_frames: i32, fade_out_frames: i32,
    fade_in_interpolation: Interpolation, fade_out_interpolation: Interpolation,
    opacity: f64,
    transform: Transform,     // 归一化 0..1 画布坐标
    crop: Crop,               // 归一化边缘裁切
    link_group_id: Option<String>,    // A/V 链接组
    caption_group_id: Option<String>, // 字幕组
    text_content: Option<String>,     // 仅 text clip
    text_style: Option<TextStyle>,
    // 关键帧轨道(每个可动画属性一条,None=无动画)
    opacity_track: Option<KeyframeTrack<f64>>,
    position_track: Option<KeyframeTrack<AnimPair>>,  // (x, y)
    scale_track:    Option<KeyframeTrack<AnimPair>>,  // (w, h)
    rotation_track: Option<KeyframeTrack<f64>>,
    crop_track:     Option<KeyframeTrack<Crop>>,
    volume_track:   Option<KeyframeTrack<f64>>,
}
```

**Clip 上挂着大量纯函数派生逻辑(必须原样移植到 Rust,这是"忠实复刻编辑算法"的核心)**:
- `end_frame = start_frame + duration_frames`(`Timeline.swift:119`)
- `source_frames_consumed = round(duration_frames * speed)`(`:122`)——速度与时长的耦合关系
- `opacity_at(frame)` / `volume_at(frame)` / `transform_at(frame)` / `crop_at(frame)`:**先采样关键帧,再叠加 fade 包络,再乘静态值**(`:130-226`)。`volume_at` 的顺序是 `静态volume × 关键帧dB增益 × fade包络`(`:189-198`),关键帧存的是 dB,采样后经 `VolumeScale.linearFromDb` 转线性。
- `fade_multiplier(at:)`:取 in/out 两条斜坡的 `min`,支持 linear/smooth(`smoothstep`)(`:211-226`)。
- `timeline_frame(source_seconds:fps:)`:源秒 → 经 trim/speed/placement 投影回时间线帧(`:229-236`)——用于把转写时间码映射到时间线。

### 2.3 关键帧系统(`Models/Keyframe.swift`)
```rust
enum Interpolation { Linear, Hold, Smooth }
struct Keyframe<V> { frame: i32, value: V, interpolation_out: Interpolation }  // frame 是 clip 相对偏移!
struct KeyframeTrack<V> { keyframes: Vec<Keyframe<V>> }  // 按 frame 有序
```
- **存储用 clip 相对帧偏移,公开 API 用绝对时间线帧**——`to_offset`/`to_abs` 来回转换(`Keyframe.swift:100-102`),移植时务必保持这个约定,否则 split/move 会错位。
- 采样 `sample(at:fallback:)`(`:231-249`):端点钳制 + 区间查找 + 按左端点 `interpolation_out` 决定 hold/linear/smooth 插值。`AnimPair`/`Crop`/`f64` 都实现 `KeyframeInterpolatable`。
- `smoothstep(t) = t*t*(3-2*t)`(`:40`)。

### 2.4 几何:Transform / Crop(归一化坐标)
`Transform`(`Timeline.swift:364-498`):`centerX/Y, width, height`(全部 0..1 相对画布)、`rotation`(度,顺时针正)、`flipH/V`。带**遗留 key 迁移**(旧 `x/y` topLeft → 新 `centerX/Y`,`:419-442`),并内置画布吸附(`snapToCanvasEdges`/`snapCenterToCanvasCenter`)。`Crop`(`:501-510`)是四边归一化内缩。

### 2.5 媒体资产模型(运行时 vs 持久化分离 —— 重要设计)

**这是上游一个关键架构决策,OpenTake 必须照搬:**

| 角色 | 类型 | 说明 | 证据 |
|---|---|---|---|
| 运行时富对象 | `MediaAsset`(`@Observable class`,`Models/MediaAsset.swift`) | 内存中,持有 `NSImage 缩略图`、解码出的 `sourceWidth/Height/FPS`、`hasAudio`、生成状态。**不直接序列化**,工程打开时从 manifest 重建 | `MediaAsset.swift:4-62` |
| 磁盘条目 | `MediaManifestEntry`(`struct Codable`,`Models/MediaManifest.swift`) | 只存可序列化字段:id、name、type、`source`、duration、生成参数、源尺寸 | `MediaManifest.swift:20-34` |
| 路径抽象 | `MediaSource` enum | `.external(absolutePath)` 或 `.project(relativePath)`——**clip 永不存路径,只存 `media_ref` (= asset id)** | `MediaManifest.swift:65-68` |

资产组织成文件夹树:`MediaFolder { id, name, parent_folder_id }`(`Models/MediaFolder.swift`),manifest 顶层是 `{ version, entries[], folders[] }`。

### 2.6 持久化:`.palmier` 是一个目录包(file package),不是单文件

`Project/VideoProject.swift` + `Utilities/Constants.swift:104-121` 给出完整格式:

```
MyProject.palmier/                    # UTI: io.palmier.project, 扩展名 palmier
├── project.json        # Timeline(JSONEncoder 直接编码)         ← Project.timelineFilename
├── media.json          # MediaManifest(entries + folders)        ← Project.manifestFilename
├── generation-log.json # GenerationLog(AI 生成审计,append-only)  ← Project.generationLogFilename

├── media/              # 内部媒体文件(.project 相对路径指向这里)
└── chat-sessions/      # Agent 对话历史,每会话一个 <uuid>.json
```

- **读**(`VideoProject.read:31-64`):off-main 解码三个 JSON → 暂存 → `makeWindowControllers` 时灌进 `editorViewModel` 并从 manifest 重建 `MediaAsset`(`restoreAssetsFromManifest:304-339`,重建时异步生成波形/缩略图/元数据)。

- **autosave-in-place 开启**(`:29`),`updateChangeCount` 桥接 dirty 状态到 store(`:130-138`)。
- **所有 Codable 都写了容错 decode**(`init(from:)` 对每个字段 `try?` + 默认值,见 `Clip:330-361`、`Transform:419`、`TextStyle:50`、`MediaManifest:8`)——**这是工程文件向后兼容的核心手法,Rust 端用 `#[serde(default)]` + `Option` 等价实现。**

> **OpenTake 持久化建议**:沿用"目录包"格式(Tauri 下就是普通目录,跨平台无 file-package 概念,反而更简单)。`project.json` schema 与上游保持字段级一致,这样上游导出的工程理论上可被 OpenTake 读取(便于迁移与对拍测试)。`PalmierProjectExporter`(`Export/PalmierProjectExporter.swift`)是"收集所有外部媒体进包、改写为相对路径"的归档器,直接照搬逻辑。

## 3. 关键数据流

### 3.1 编辑写路径:命令 → 不可变 swap → 撤销注册 → 投影失效

**所有编辑都遵循同一个事务模式(`EditorViewModel+ClipMutations.swift`),这是整个系统的心脏:**

核心原语 `withTimelineSwap(actionName:_:)`(`ClipMutations.swift:240-252`):
```
let before = timeline                    // 整树值拷贝(struct,廉价)
undoManager.disableUndoRegistration()
work()                                   // 任意复杂的就地修改
undoManager.enableUndoRegistration()
let after = timeline
guard before != after else { return }    // Equatable 短路:没变就不记 undo
registerTimelineSwap(undo: before, redo: after, actionName)  // 注册整树 swap
notifyTimelineChanged()                  // 触发预览重建
```
`registerTimelineSwap`(`:230-237`)用**双向自注册**实现 redo:undo 闭包里把 `timeline = before` 后,再注册一个 `undo: after, redo: before` 的反向 swap。这是 NSUndoManager 的经典手法。

三种粒度的撤销(都同一思路,粒度递减):
1. **整树 swap**(`withTimelineSwap`)——结构性大改(add/move/ripple/clearRegion),per-clip undo 无法干净组合时用。
2. **多 clip 状态 swap**(`mutateClips:284-318`)——捕获若干 clip 的 before/after 快照。
3. **单 clip 属性 swap**(`commitClipProperty:400-428`)——拖拽控件,配 `dragBefore` 快照 + `debouncedCommitClipProperty`(连续控件如 ColorPicker 静默 400ms 后落一次 undo,`:353-368`)。

**编辑算法都抽成了纯函数 enum**(无副作用,极易移植 + 单测):
- `OverwriteEngine.computeOverwrite(clips, start, end) -> [Action]`(`Editor/OverwriteEngine.swift`):覆盖式落 clip 时,算出对重叠 clip 该 remove/trimEnd/trimStart/split。
- `RippleEngine`(`Editor/RippleEngine.swift`):`computeRippleShifts`(删除后收缩)、`computeRipplePush`(插入推开)、`computeRippleShiftsForRanges`(给 sync-locked 轨道按外部 range 收缩)、`mergeRanges`。
- ripple delete 的**拒绝语义**(`EditorViewModel+Ripple.swift:38-70`):若任一 sync-locked 跟随轨道收缩后会碰撞或越过 0,整个操作 `refuseRipple`(beep + log)而非破坏对齐——这是专业 NLE 的关键行为,必须复刻。
- split(`splitSingleClip:118-164`)的精髓:`leftSource = round(splitOffset * speed)`,左右半各自重算 trim;**6 条关键帧轨道在切点各插一个边界关键帧保持曲线连续**(`splitKeyframeTrack:167-187`),而不是把整条 track 复制给两半。

A/V 链接(`EditorViewModel+Linking.swift`):同 `link_group_id` 的 clip 作为一个单位 move/trim/split/delete。`partnerMoves`、`timingPropagationPartners`、`linkGroupOffsets`(检测失同步)全是纯函数。

### 3.2 读路径(投影到预览/导出)—— **这是唯一与 AVFoundation 强耦合、OpenTake 要换 FFmpeg 的地方**

`Preview/CompositionBuilder.swift`(843 行,核心)把不可变 `Timeline` **投影**成 `AVMutableComposition + AVAudioMix + AVVideoComposition`:
- 每条 timeline 轨道 → 一条 composition track;clip 按 `start_frame` 插入,`insertEmptyTimeRange` 填间隙(`insertClip:301-344`)。
- **图片/Lottie 先转成静帧视频**(`ImageVideoGenerator.stillVideo` / `LottieVideoGenerator.lottieVideo`,带缓存),因为 AVFoundation 只吃视频轨——FFmpeg 路径同理需要把图片/文字/Lottie 物化成可合成的源。
- speed≠1 → 单独 track + `scaleTimeRange`。
- 底部铺一条黑底视频覆盖整个时长(`insertBlackBackground:346-372`)。
- **`buildVisuals`(`:374-461`)是关键帧 → 合成指令的转译器**:opacity/transform/crop/volume 关键帧被展开成 `AVVideoCompositionLayerInstruction` 的 ramp;smooth 段按 `smoothSegments=8` 细分成多段线性 ramp(`trackOps:780-842`、`emitTransform:617-697`)。**这套"关键帧 → 分段线性 ramp"算法是软件无关的**,OpenTake 在 Rust 里算出每帧/每段的属性值,交给 FFmpeg filtergraph(如 `overlay`/`scale`/`rotate`/`colorchannelmixer`/`volume` + `sendcmd`/表达式)即可。
- **文字不进 composition**:预览期用 `CATextLayer` 覆盖(`Preview/TextLayerController.swift`),导出期用 `AVVideoCompositionCoreAnimationTool` 烤进视频(`ExportService.swift:237-248`)。OpenTake 对应:预览用前端 DOM/Canvas 叠字,导出用 FFmpeg `drawtext`/`subtitles` 或离屏渲染 PNG 序列叠加。

预览播放引擎 `Preview/VideoEngine.swift`:包 `AVPlayer`;`rebuild()`(`:137-185`)异步重建 composition 后 `replaceCurrentItem`;`refreshVisuals()`(`:187-205`)只重算 audioMix/videoComposition(属性改动走这条快路,不重建轨道);交互式 scrub 有 30fps 节流 + 容差自适应(`:225-272`)。`EditorViewModel.notifyTimelineChanged()`(`:304-312`)与 `notifyTimelineChangedDebounced`(`:315-324`)是 store → 引擎的失效信号。

导出 `Export/ExportService.swift`:复用同一个 `CompositionBuilder.build`,换 `AVAssetExportSession`(H.264/H.265/ProRes)或走 `XMLExporter`(FCPXML 互通,不渲染)。**OpenTake:`CompositionBuilder` 的等价物拆成两个 Rust 后端 —— 预览用低延迟管线(FFmpeg → 纹理/逐帧),导出用 FFmpeg 全质量编码;两者共享同一个"Timeline → 渲染计划(RenderPlan)"的纯函数。**

### 3.3 撤销/重做小结
基于 `NSUndoManager` 的三粒度双向 swap(见 3.1)。**OpenTake 在 Rust 端自建 `UndoStack`**:因为整个 `Timeline` 是廉价可克隆的值类型,最简单可靠的做法就是**整树快照栈**(`Vec<(Timeline, action_name)>`)——上游 `withTimelineSwap` 本质就是这个,只是借了 NSUndoManager 的双向注册。前端不持有撤销逻辑,只发 `undo`/`redo` 命令。

**依赖法则(经代码验证):**
- `Models/*` 是**最底层叶子**,零依赖,纯值类型 + 纯派生函数。
- 纯算法引擎(`OverwriteEngine`/`RippleEngine`/`SnapEngine`)只依赖 `Models`,是 `enum` 静态函数,**完全无状态、可直接单测、可直接移植 Rust**。
- `EditorViewModel` 是**中枢 hub**:所有命令、撤销、选择、媒体库都在这里(通过 ~20 个 extension 文件切分,但仍是一个 `@MainActor` 类)。它依赖 Models + 引擎 + Services + VideoEngine。
- **UI 是纯消费者**:`EditorView` 用 `@Environment(EditorViewModel)` 注入,五面板用 `NSSplitViewController` 嵌套(`EditorView.swift:38-203`,三种 `LayoutPreset`)。UI 读 store + 发命令,不持有领域逻辑。
- **AI 双入口、单执行器**:`AgentService`(应用内,`Agent/AgentService.swift:441` 调 `executor.execute`)和 `MCPService`(外部,`MCP/MCPService.swift:92` 调 `toolExecutor.execute`)**都把请求归一成 `ToolExecutor.execute(name, args)`**,后者直接操作 `EditorViewModel`(`ToolExecutor.swift:22-70`)。**这是整个架构对 OpenTake 最重要的一条**:编辑能力有唯一的"命令层",UI、应用内 Agent、外部 MCP 三者是这层的三个对等客户端。

## 5. Rust core / React 前端 / Tauri command 边界 切分建议

### 5.1 总原则(基于上游证据推导)
上游把"领域模型 + 编辑算法 + 撤销 + 投影计划"写成了**几乎纯函数式、值语义**的形态(Models 全 struct、引擎全 enum 静态函数、撤销靠整树 diff)。**这部分 100% 应下沉到 Rust core,且能近乎机械地移植。** 上游唯一"脏"的耦合是 `CompositionBuilder → AVFoundation` 和 UI 的 `@MainActor` 单线程 store——前者换 FFmpeg,后者拆成 Rust 真相源 + 前端镜像。

### 5.3 接口草案(关键)

**(a) 唯一编辑命令枚举(= 上游 `ToolName` + UI 动作的并集,见 `ToolExecutor.swift:73-105`)**
UI 手势、应用内 Agent、外部 MCP 三者**全部归一到这一个 enum**,这样撤销/校验/遥测只需写一遍:
```rust
#[derive(Serialize, Deserialize)]
#[serde(tag = "type", rename_all = "snake_case")]
enum EditCommand {
    AddClips { assets: Vec<String>, track_index: usize, start_frame: i32, segments: HashMap<String, (f64,f64)> },
    InsertClips { /* ripple insert */ },
    MoveClips { moves: Vec<ClipMove> },
    RemoveClips { ids: Vec<String> },
    SplitClip { clip_id: String, at_frame: i32 },
    TrimClips { edits: Vec<TrimEdit> },
    SetClipProperties { clip_id: String, patch: ClipPatch },
    SetKeyframes { clip_id: String, property: AnimatableProperty, keyframes: Vec<KeyframeDTO> },
    RippleDeleteRanges { anchor_clip_id: String, ranges: Vec<FrameRange> },
    AddTexts { .. }, AddCaptions { .. },
    Link { ids: Vec<String> }, Unlink { ids: Vec<String> },
    RemoveTracks { .. }, CreateFolder { .. }, MoveToFolder { .. },
    Undo, Redo,
}

struct EditResult {
    changed: bool,                 // 对应上游 `timeline != before` 短路
    action_name: String,           // 进撤销栈
    affected_clip_ids: Vec<String>,
    timeline_version: u64,         // 单调递增, 前端据此决定是否重取
    summary: String,               // 给 Agent/MCP 看的文字结果
}
```
所有命令在 `command::apply` 里跑"读快照 → 改 → 若变则压 UndoStack(整树快照)→ 发 `timeline_changed` 事件"——直接对应上游 `withTimelineSwap`。

**(b) Tauri command 表面(薄)**
```rust
#[tauri::command] fn edit_apply(cmd: EditCommand, state: State<EditorState>) -> EditResult;
#[tauri::command] fn get_timeline(window: Option<FrameRange>) -> TimelineDTO; // 对应 get_timeline 工具

#[tauri::command] fn project_save() -> Result<(), String>;

#[tauri::command] fn export_start(format: ExportFormat, resolution: ExportResolution, out: String);
// 事件(Rust → 前端): timeline_changed{version}, preview_frame{...}, export_progress{p}
```

**(c) 关键边界决策(踩坑预防,均有上游对应)**
1. **真相源在 Rust,前端只持镜像。** 上游 store 是单实例所以 UI 与 MCP 天然一致;OpenTake 跨进程,**必须让 Rust 持有权威 `Timeline`,前端拿快照 + 版本号**。每次 `edit_apply` 返回新版本号并广播 `timeline_changed`,前端据此增量或全量重取(上游靠 `timelineRenderRevision &+= 1`,`EditorViewModel.swift:27-30`,可照搬为 `u64` 版本)。
2. **撤销栈在 Rust。** 整树快照栈(`Timeline` derive `Clone`),不要试图在前端做撤销。
3. **MCP server 内嵌 Rust core,直连 `command::apply`**,与 Tauri command 平级——**不要让 MCP 走前端**。上游 `MCPService` 和 UI 共享同一 `ToolExecutor`(`MCPService.swift:31-32`),OpenTake 对应:MCP server 与 Tauri 后端共享同一 `EditorState`。MCP 工具 schema 直接复用 `Agent/Tools/ToolDefinitions.swift`(652 行,已是完整的 JSON Schema 定义)。注意上游 MCP 有"只撤销 AI 自己的编辑"的安全栈(`ToolExecutor.swift:109-123`),值得复刻。
4. **预览/导出共享一个纯函数 `RenderPlan`,两个 FFmpeg 后端。** 把上游 `CompositionBuilder.buildVisuals`(关键帧→分段线性 ramp,`smoothSegments=8`)移植成 Rust 纯函数 `Timeline -> RenderPlan{ per_clip ramps }`;预览后端追求低延迟(seek + 单帧),导出后端追求质量(整段编码)。**这是把"算法"与"FFmpeg 执行"解耦的关键,也保证预览与导出像素一致。**
5. **媒体物化策略照搬:** 图片/文字/Lottie 在合成前先物化(上游 `ImageVideoGenerator`/`LottieVideoGenerator` 带 `media_ref` 缓存)。OpenTake 用 content-hash 缓存(图片→静帧片段、文字→PNG/drawtext、Lottie→预渲染序列)。
6. **帧↔像素换算放前端,帧↔秒换算放 Rust 边界。** 时间线缩放(`zoomScale = pixelsPerFrame`)是纯 UI 态,留前端(上游在 `TimelineGeometry`/`Constants.swift:61`)。源秒↔帧(import/转写映射,`Clip.timeline_frame`)在 Rust。
7. **容错反序列化从第一天就上 `#[serde(default)]`。** 上游每个模型的手写容错 decode(`Clip:330`、`Transform:419`、`TextStyle:50`)是工程文件长期可用的保障,Rust 用 `#[serde(default)]` + `Option<T>` 等价实现,务必同步字段语义以便与上游工程对拍。
