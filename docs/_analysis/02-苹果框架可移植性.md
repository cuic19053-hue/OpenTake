# OpenTake Rust Core 可行性评估：上游 Apple 媒体框架使用全貌与移植攻坚清单

> 基于对 `palmier-pro-upstream/Sources/PalmierPro`(209 个 Swift 文件)的全量符号级搜索与核心引擎文件逐行阅读。结论先行:**上游媒体引擎几乎完全建立在 AVFoundation 的"声明式合成模型"之上(`AVComposition` + `AVVideoComposition` + layer instructions + ramps),没有任何一帧是手动逐像素渲染的。这正是移植到 FFmpeg/wgpu 的最大风险点 —— 你要替换的不是"几个 API 调用",而是一整套由 Apple 实现的、带关键帧插值语义的合成器。**

关键全局事实(符号级证据):
- 直接 `import`:AVFoundation 24 文件、AppKit 62、SwiftUI 78、CoreVideo 4、CoreText 2、CoreGraphics 6、Speech 1、CoreML 2、Accelerate 2、ImageIO 4。
- **零直接使用** Metal / MetalKit / VideoToolbox / Vision / CoreImage(`CIFilter`/`CIContext`/`CIImage`)/ MetalPerformanceShaders / AVKit / CoreMedia(显式 import)。grep `MTLDevice|CIFilter|VTCompressionSession|VNRequest|MTKView` 全空。这意味着**没有自定义 GPU shader、没有 CoreImage 滤镜链**需要复刻 —— 所有像素级合成都委托给了 AVFoundation 黑盒。
- 全仓仅 353 行引用 `AV*/CM*/CV*` 符号,媒体引擎高度集中在 `Preview/` 与 `Export/` 两个目录。

## 1. AVFoundation 各能力的实际使用 + 文件路径证据

### 1.1 合成(Composition)—— 引擎核心,最难
**`Preview/CompositionBuilder.swift`(843 行,绝对核心)**。`enum CompositionBuilder.build(...)` 把 `Timeline` 领域模型翻译成 `AVMutableComposition`:
- 逐 track 逐 clip `addMutableTrack` + `insertTimeRange(_:of:at:)`(L326),gap 用 `insertEmptyTimeRange`(L316)。
- **变速**:`scaleTimeRange(_:toDuration:)`(L339)—— 改变 source 区间长度后缩放,音视频都走这条路。
- **音频混合**:`AVMutableAudioMix` + `AVMutableAudioMixInputParameters.setVolumeRamp` 发射逐段线性音量包络(L487/L501),静音/淡入淡出/关键帧音量全部折叠进 `volumeAt(frame:)`。
- **视频可视属性**:`buildVisuals(...)`(L375)用 `AVVideoCompositionLayerInstruction.Configuration` 发射:
  - `setOpacity` / `addOpacityRamp`(L407/L585)
  - `setTransform` / `addTransformRamp`(L632/L680)—— 关键帧位置/缩放/旋转
  - `setCropRectangle` / `addCropRectangleRamp`(L724/L725)—— 裁剪
- **关键帧插值语义自己实现**:`smooth` 段用 8 段 `smoothstep` 细分(L464/L824),`hold`/`linear`/`smooth` 三种 `interpolationOut`,并用分数 `CMTime` 避免整帧取整塌缩(L676)。这是**忠实复刻上游编辑算法的关键所在**。
- **底层黑底**:`insertBlackBackground` 用一段生成的黑视频做最底层(L346)。
- 色彩管线锁定 BT.709(L456-458)。

驱动方:**`Preview/VideoEngine.swift`(326 行)** `rebuild()`(L137)在后台 Task 调 `CompositionBuilder.build`,产出 `AVPlayerItem(asset:)` 并挂 `audioMix`/`videoComposition`(L176-179)。`refreshVisuals()`(L187)做"只改可视属性不重建轨道"的快路径。

## 2. 能力 → OpenTake 栈映射 + 风险/成熟度判断

| 能力 | 上游实现 | OpenTake 替代 | 风险 | 成熟度判断 |
|---|---|---|---|---|
| **合成器(轨道/ramp/transform/crop 关键帧)** | `CompositionBuilder` + `AVVideoComposition` layer instructions | **wgpu 自写帧合成器**(每帧:解码→采样关键帧→变换/裁剪/混色→合成);ramp 语义在 Rust 侧重算(已有 `smoothstep`/hold/linear 公式可直接照搬 L824) | **极高 / blocker #1** | **无现成替代**。AVFoundation 帮你做了"按 instruction 逐帧合成"的全部脏活。FFmpeg `filter_complex`(overlay/scale/rotate/colorchannelmixer + `sendcmd`/表达式)只能近似且不支持任意关键帧曲线。**必须自建 wgpu compositor**。这是整个 Rust core 可行性的命门。 |
| **解码/读帧** | `AVAssetReader` | **ffmpeg-next**(libavcodec/libavformat) | 中 | **成熟**。ffmpeg-next 绑定稳定,VideoToolbox/NVDEC/VAAPI 硬解可用。 |
| **编码/导出** | `AVAssetExportSession` / `AVAssetWriter` | **ffmpeg-next**(libx264/libx265/ProRes encoder) | 中 | **成熟**,但预设/码率/色彩需逐项对齐(上游预设是黑盒 preset,需测试逼近)。 |
| **播放** | `AVPlayer` + `AVPlayerLayer` | **自建播放循环**:ffmpeg 解码 → wgpu 上屏 + cpal 音频,A/V 同步与 seek 自己写 | **高 / blocker #2** | **无开箱即用**。`AVPlayer` 的精确 seek + tolerance + 实时合成预览是巨大隐性工作量。Rust 无对等物(`gstreamer-rs` 是最接近的,但引入 GStreamer 与"wgpu 合成"路线冲突)。 |
| **图片→视频 hack** | `ImageVideoGenerator` 烧静止视频 | **不需要移植** —— wgpu compositor 里图片本就是一张纹理,直接当帧源 | 低(架构上消失) | 自建合成器后此 hack 整类删除,是简化点。 |
| **Alpha 预乘** | AVAssetReader + vImage | wgpu 着色器里直接处理 premultiplied alpha,或 ffmpeg `premultiply` 滤镜 | 低 | 合成器内建,trivial。 |
| **缩略图** | `AVAssetImageGenerator` | **ffmpeg-next** seek+解一帧 + image crate 缩放 | 低 | **成熟**,机械工作。sprite 网格缓存逻辑(`MediaVisualCache`)可原样移植(纯 CGContext→可换 `tiny-skia`/`image`)。 |
| **波形** | `DSWaveformImage` | **Symphonia** 解码 PCM + 自算 RMS/peak 降采样 | 低 | **成熟**。Symphonia 纯 Rust 解大多数音频;降采样算法上游已明确(归一化 0..1,count 见 L186)。 |
| **字幕/文字渲染** | `CATextLayer` + `AVVideoCompositionCoreAnimationTool` | **Skia(skia-safe)或 cosmic-text + Vello** 排版渲染成纹理,喂给 wgpu compositor | **高 / blocker #3** | **可行但工程量大**。`CATextLayer` 的换行/对齐/阴影/描边/背景盒/discrete-opacity 动画要逐项复刻。推荐 **cosmic-text(排版)+ tiny-skia/Vello(光栅化)**;字体注册用 `fontdb`。**字幕的逐帧 opacity + 与视频合成必须落到合成器里**,不能再靠 Apple 的 animation tool。 |
| **on-device 转写** | macOS 26 `SpeechAnalyzer`/`SpeechTranscriber` | **whisper.cpp**(whisper-rs) | 中 | **成熟但语义不同**。上游用的是**全新 macOS 26 API**(非 `SFSpeechRecognizer`),输出 segment + 逐 token 时间码(`audioTimeRange`,L309)。whisper.cpp 给 word/segment 时间戳能力相当甚至更跨平台,但**多语言模型选择、标点/etiquette 替换、locale 自动匹配**需在 Rust 侧重建。`TranscriptionResult`/`TranscriptionWord`/`TranscriptionSegment` 数据模型可原样保留。 |
| **视觉语义搜索(CLIP)** | CoreML 跑 `siglip2-base-patch16-256`(768 维)+ swift-transformers tokenizer | **candle 或 ort(onnxruntime)** 跑 SigLIP2 + tokenizers(HF Rust 库) | 中 | **成熟**。`Search/Models/VisualEmbedder.swift` 逻辑清晰(squash-resize 到方形 L84、image/text 双编码器)。模型托管在 HF(`palmier-io/siglip2-base-coreml`),需转 ONNX/GGUF。`tokenizers` crate 直接对应 swift-transformers。 |
| **Lottie** | `lottie-ios`(airbnb)烧 ProRes4444 | **`rlottie` FFI 或 `velato`(Vello+Lottie)** | 中 | rlottie(C++,Samsung)成熟可 FFI;velato 纯 Rust 但覆盖度有限。同样,渲成纹理交给合成器,不再烧中间视频。 |
| **字体注册** | `CTFontManagerRegisterFontURLs` | **`fontdb` + `cosmic-text`** | 低 | 成熟。`BundledFonts.swift` 的家族枚举/可预览判断逻辑可移植。 |

## 3. 文本/字体/图形渲染(CoreText / CoreImage / Metal / CoreGraphics)用在哪、如何替换

- **CoreText**(2 文件):`Utilities/BundledFonts.swift`(字体注册 `CTFontManagerRegisterFontURLs`、家族枚举)与 `Agent/Tools/OverviewRenderer.swift`(`CTLineCreateWithAttributedString`/`CTLineDraw` 在缩略图上烧时间戳,L98-103)。→ **fontdb + cosmic-text + tiny-skia**。
- **CoreGraphics**(6 文件):全部是 `CGContext` 离屏绘制(sprite 网格 `MediaVisualCache` L270、storyboard `OverviewRenderer` L74、CLIP 预处理 `VisualEmbedder` L72、文字快照合成 `EditorViewModel+MediaLibrary` L414 `textRoot.render(in:)`)。→ **tiny-skia(2D 光栅)或 image crate**;`CALayer.render(in:)` 这类把图层树拍平的操作,在 Rust 侧改为"合成器直接渲染该帧"。
- **CoreImage**:**完全没用**。无滤镜链可复刻 —— 利好。
- **Metal**:**完全没用**(没有自定义 shader)。但这也意味着**上游把 GPU 工作全交给了 AVFoundation**;OpenTake 要自己用 wgpu 把这块补回来。
- **CoreVideo**(4 文件):`CVPixelBuffer` 作为 reader/writer/CoreML 的像素载体。→ wgpu texture / ffmpeg `AVFrame` / `ndarray`,按场景替换。
- **Accelerate / vImage**(2 文件):alpha 预乘 + 向量搜索点积。→ 着色器内做 / `ndarray`+SIMD。
- **ImageIO**(4 文件):`CGImageSource` 读图元数据/缩略图、`CGImageDestination` 写 JPEG。→ **image crate**(+ `kamadak-exif` 取朝向)。

## 4. 移植攻坚清单(最难 → 最易)+ 落地建议

**🔴 1. wgpu 帧合成器(blocker,决定项目成败)**
复刻 `CompositionBuilder.buildVisuals` 的全部语义:多轨叠放、opacity/transform/crop 关键帧、smooth(8 段 smoothstep)/hold/linear 插值、归一化画布坐标 → 像素仿射(`affineTransform` L599 公式可直接照搬)、BT.709。
*落地*:先在 Rust 重建 `Timeline`→"每帧合成指令"的纯逻辑层(可单元测试、与渲染解耦,直接移植 `trackOps`/`emitTransform` 算法),再写 wgpu render graph 逐帧执行。**这是唯一无法外包、必须从零的部分,建议最先做 PoC。**

**🔴 2. 播放/预览引擎(blocker)**
ffmpeg 解码 + wgpu 上屏 + cpal 音频,自写 A/V 同步、精确 seek、scrub 节流(`VideoEngine` L225-272 的节流器逻辑可移植)。
*落地*:复用合成器输出;seek tolerance 概念在自建管线里用"解码到最近关键帧 + 丢帧到目标"实现。预览分辨率降档以保帧率。

**🟠 3. 文字/字幕渲染管线**
cosmic-text 排版 + tiny-skia/Vello 光栅 → 纹理 → 合成器叠加;逐帧 opacity 自己驱动(替代 `CAKeyframeAnimation` discrete)。复刻阴影/描边/背景盒/对齐/换行。
*落地*:`TextStyle`/`TextLayout` 数据模型保留;先做静态渲染对齐 Apple 输出(像素 diff),再接关键帧。

**🟠 4. 编码/导出对齐**
ffmpeg-next 替 `AVAssetExportSession`,逐预设(H.264/H.265/ProRes × 三档分辨率)调码率/profile/色彩逼近上游。
*落地*:`ExportFormat`/`ExportResolution` 枚举与 `renderSize` 取偶数逻辑(L39)原样保留;导出帧来自合成器。

**🟡 5. on-device 转写(whisper.cpp)**
whisper-rs 出 word/segment 时间戳;重建 locale 匹配、标点、profanity 选项。
*落地*:`TranscriptionResult`/`offsetting(by:)`(L26)等数据模型完全复用;音频抽取改 Symphonia。

**🟡 6. CLIP 语义搜索(candle/ort)**
SigLIP2 转 ONNX + tokenizers crate;`VisualEmbedder` 双编码器逻辑、squash-resize 预处理照搬。
*落地*:索引/查询/向量存储(`Search/`)大多是纯逻辑,易移植;只换推理后端。

**🟡 7. Lottie**
rlottie FFI(优先,覆盖度高)或 velato;渲成纹理交合成器。
*落地*:`LottieVideoGenerator` 的"烧中间视频"整类删除。

**🟢 8. 缩略图 + sprite 缓存**
ffmpeg seek 解帧 + image 缩放;`MediaVisualCache` 的 sprite 网格 + mtime 缓存键(L209)逻辑原样移植。

**🟢 9. 波形(Symphonia)**
解 PCM + RMS 降采样;归一化与 count 公式(L186)照搬;`.waveform` 二进制缓存格式保留。

**🟢 10. 字体注册 + 图像编解码**
fontdb 替 CTFontManager;image crate 替 ImageIO。机械工作。

### 一句话结论
**项目可行,但"可行性"几乎全部押在自建 wgpu 合成器 + 自建播放引擎这两件事上。** 上游 80% 的"媒体代码量"(reader/writer/缩略图/波形/转写/CLIP/字体)都有成熟 Rust 替代、可机械移植;但被 AVFoundation 黑盒掉的 20%(声明式合成 + 播放协调)是没有任何现成替代品、且承载着"忠实复刻编辑算法"承诺的核心。**建议第一步就做合成器 PoC(单轨视频 + 一个 transform 关键帧 + 一条字幕,导出帧与上游做像素 diff),用它来验证整条 Rust core 路线,再铺开其余。**
